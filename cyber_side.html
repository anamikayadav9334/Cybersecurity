Origin & Core Concept
Developed by MITRE Corporation in 2013 as an internal project (FMX — Fort Meade Experiment), observing attacker behaviour on a sandboxed network. Made public in 2015. It answers: "What do attackers actually do — step by step — once they're inside?" Unlike CVE (vulnerabilities) or NIST CSF (governance), ATT&CK focuses on post-compromise behaviour and attacker tradecraft. It is a living knowledge base — updated ~twice yearly from real incidents and threat reports.
Three matrices
14
Enterprise
~200 techniques · ~400 sub-techniques. Covers Windows, macOS, Linux, Cloud (AWS/Azure/GCP), SaaS, Containers. Most widely used matrix.
14
Mobile
iOS & Android. Covers app-based attacks, network-based, physical access, MDM bypass, spyware (Pegasus-style).
12
ICS
Industrial control systems — SCADA, PLCs, HMIs. Power, water, oil & gas, manufacturing OT. Ref: Stuxnet, Industroyer, Triton.
Key vocabulary
TTP
Tactics (goal) + Techniques (how) + Procedures (specific real-world instance). TTPs are the gold standard of threat intel — harder to change than IPs or hashes.
Procedure
A concrete, real-world example of a technique used by a specific group. e.g. "APT29 used PowerShell to download a payload from a C2 server."
Mitigation
ATT&CK maps each technique to recommended defensive measures (MFA, credential hardening, network segmentation, etc.) via M#### IDs.
Data Source
What telemetry you need to detect a technique — process logs, network flow, registry events, file system activity, Windows Event IDs.
What it is NOT
Not a compliance framework — no certification, no mandate, no audit checkboxes.
Not a linear kill chain — attackers skip, repeat, or reorder phases. It's a menu of behaviours, not a sequence.
Not a vulnerability database — that's CVE/NVD. ATT&CK covers behaviour regardless of specific CVE.
Not a detection tool — it's a knowledge base. Detection engineering is built on top (Sigma rules, SIEM queries).
What are tactics?
Tactics represent the adversary's objective at each phase — they answer "Why is the attacker doing this?". The 14 Enterprise tactics follow a rough attacker progression from pre-compromise to impact. They are the columns of the ATT&CK matrix; techniques are the rows underneath each column.

Structure: Technique → Sub-technique
Each tactic has multiple Techniques (T####) — the how. Each technique can have Sub-techniques (T####.001) for specific variants. Every technique page includes: Mitigations, Data Sources, Detection guidance, and real Procedure Examples cited from threat reports.
Most-used techniques (high attacker prevalence)
T1059 — Command & Scripting Interpreter
Attacker runs code via PowerShell (.001), Bash (.004), Python (.006), VBScript (.005), Windows Cmd (.003). Used by nearly every APT. Sub-techniques make detection nuanced — each interpreter has different telemetry.
T1078 — Valid Accounts
Stealing and reusing legitimate credentials to blend in with normal activity. Bypasses perimeter controls because access looks authorized. Why MFA alone is insufficient — attackers steal session tokens.
T1566 — Phishing
Most common initial access technique globally. Sub-techniques: spearphish attachment (.001), spearphish link (.002), spearphish via service (.003). Enables malware delivery or credential harvesting.
T1055 — Process Injection
Inject code into a legitimate process (explorer.exe, svchost.exe) to hide malicious activity and inherit its privileges. Evades AV/EDR tools that trust whitelisted processes.
T1027 — Obfuscated Files/Info
Encode/encrypt/compress payloads to evade signature-based AV. Techniques: Base64 encoding, XOR encryption, steganography, packed executables. Paired with T1059 constantly.
T1486 — Data Encrypted for Impact
The ransomware technique. Attacker encrypts files and demands payment for decryption key. Requires successful prior stages: initial access → lateral movement → privilege escalation before running.
T1110 — Brute Force
Credential stuffing (.004), password spraying (.003), online guessing (.001). Password spray avoids account lockouts by trying one password against many accounts.
T1003 — OS Credential Dumping
Extract credentials from LSASS memory, SAM database, NTDS.dit. Mimikatz is the canonical tool. Enables lateral movement via pass-the-hash or direct password reuse.
Detection components on every technique page
Data Sources
What telemetry to collect — e.g. "Process creation logs", "Windows Event ID 4688", "PowerShell Script Block Logging (EID 4104)", "Network flow", "Registry modification".
Detection Notes
Guidance on anomalies to look for. Not specific SIEM rules — the logic behind them. Pair with Sigma rule library for concrete queries.
Mitigations (M####)
Each technique links to structured Mitigations. e.g. M1032 = Multi-factor Authentication, M1026 = Privileged Account Management, M1030 = Network Segmentation.
Procedure Examples
Real named threat groups that used this technique — links to campaigns and software, cited from public threat reports (Mandiant, Crowdstrike, NSA advisories).

Software catalogue (S#### IDs) — tools & malware
Cobalt Strike S0154
Commercial pen-test tool heavily abused by 100+ threat groups. Implements C2 beaconing, lateral movement, credential dumping, process injection. The "Kali Linux of APTs".
Mimikatz S0002
Credential dumping tool. Extracts plaintext passwords, NTLM hashes, Kerberos tickets from Windows LSASS memory. Essential to nearly every lateral movement operation.
BloodHound S0521
Active Directory attack path mapper. Used by both red teams and attackers to find the shortest path to Domain Admin through AD relationships and permissions.
Emotet S0367
Modular malware dropper / botnet. Delivered Ryuk and Conti ransomware payloads. Spreads via malicious macros in email attachments. Taken down 2021, returned 2022.
Campaigns (C#### IDs)
Track named real-world intrusion events — linking groups, software, and techniques to a specific incident with a timeline. Example: C0024 SolarWinds Compromise — maps every technique APT29 used end-to-end, from supply chain poisoning (T1195) to cloud persistence (T1078). Useful for understanding full attacker kill chains.
Red Team / Adversary Emulation
Build realistic attack simulations that mimic real threat actors — not generic pen tests. ATT&CK IDs make scope and findings precise and structured.
  • Select target threat group (e.g. APT29)
  • Export their technique list from ATT&CK
  • Build emulation plan step-by-step (MITRE publishes free plans)
  • Execute — test whether blue team detects each technique
  • Report by ATT&CK ID for structured, comparable findings
Blue Team / Detection Engineering
Map your existing SIEM/EDR detections to ATT&CK to find blind spots. Coverage gaps drive your defensive roadmap.
  • List every SIEM/EDR alert and tag it with a technique ID
  • Visualise in ATT&CK Navigator — uncovered cells = gaps
  • Prioritise gaps by threat group relevance to your sector
  • Write Sigma rules / SIEM queries to fill high-priority gaps
Purple Team
Red & blue team working together — red executes a technique, blue validates detection in real-time. ATT&CK provides the shared vocabulary so both teams talk the same language. Output: measured coverage percentage per technique.
Threat Intelligence
Structured tagging of threat reports with ATT&CK IDs makes intel actionable and comparable across teams and vendors.
  • Analyst reads threat report → tags each behaviour with T#### ID
  • Feed into SIEM/SOAR for automated detection rules
  • Compare threat actor profiles — which groups target your sector?
  • Share intel in STIX format across orgs / ISACs
SOC Benchmarking
Report to leadership: "We can detect 68% of Lazarus Group techniques. Q3 goal: 80%." Turns abstract security posture into a measurable, communicable metric. Also used during SOC triage to answer: "what phase is the attacker in? what comes next?"
ATT&CK Navigator
Free web app at mitre-attack.github.io/attack-navigator. Lets you create, colour, and layer ATT&CK matrices. Core use: visualise what you detect vs what a threat actor uses — the overlap gap drives your priorities.

STIX / TAXII — machine-readable format
Full ATT&CK knowledge base available as STIX 2.1 JSON (Structured Threat Information Expression). Ingested by SIEMs (Splunk, Elastic), SOAR platforms, and threat intel tools. TAXII servers allow automated syncing of updated ATT&CK data. This is how vendors like CrowdStrike, Microsoft Defender, and SentinelOne auto-map alerts to ATT&CK IDs.
Companion tools & integrations
Sigma Rules
Community SIEM detection rules, each tagged with ATT&CK technique IDs. Translatable to Splunk, Elastic, QRadar, Chronicle, Microsoft Sentinel, etc. 3,000+ rules open-source.
MITRE D3FEND
Companion knowledge base of defensive techniques — mapped to ATT&CK offence. "They do T1059 (PowerShell), so apply D3-PSA (Process Spawn Analysis)". Mirror image of ATT&CK.
Atomic Red Team
Red Canary's open-source test library — atomic tests for each ATT&CK technique. Run a test, verify your detection actually works. Integrates with Navigator for coverage tracking.
CTID Emulation Plans
MITRE's Center for Threat-Informed Defense publishes free full adversary emulation plans (APT29, FIN6, menuPass) ready to execute with step-by-step playbooks.
These frameworks are complementary, not competing. Most mature security programs use all of them together — each answers a different question. Think of them as layered from governance → operations → implementation.

Preventive controls stop threats before they occur. They are the first line of defence — reducing the attack surface and making it harder for adversaries to gain a foothold.
🛡️ Firewalls
Filters network traffic by rules — allow/deny by IP, port, protocol. Stateful inspection tracks active sessions. WAF (Web Application Firewall) inspects HTTP payloads for attack patterns like SQLi and XSS. First perimeter barrier.
Best use: Network edge; between untrusted internet and internal LAN
🔑 Access Control
MFA, RBAC, SSO, least-privilege — verify identity first, then grant only the minimum access needed. Role-based access assigns permissions by job function. Prevents unauthorized data/system access.
Best use: Cloud IAM, Active Directory, privileged account protection
🔐 Encryption
AES-256 at rest, TLS 1.3 in transit, PKI for trust. Data is mathematically unreadable without the decryption key — useless to anyone who intercepts it. Protects confidentiality even if storage is compromised.
Best use: Databases, email, APIs, storage — anywhere data leaves your perimeter
🔧 Patching & Hardening
Apply CVE patches promptly, remove unused services and open ports, enforce CIS benchmark or STIG configuration baselines. Eliminates known weaknesses attackers actively exploit. Patch management = a DiD layer.
Best use: All systems — especially internet-facing; monthly cycles minimum
🦠 AV / EDR / XDR
Antivirus uses signatures + heuristics. EDR adds live behavioral monitoring per endpoint and automated containment. XDR correlates signals across endpoint, network, identity, and cloud into one detection pipeline.
Best use: All endpoints, servers; XDR correlates across the whole estate
🔗 Segmentation
VLANs, DMZ, micro-segmentation divide the network into isolated zones. Limits blast radius — attackers who breach one zone can't freely reach others. Critical for stopping ransomware spread.
Best use: Critical assets (OT/SCADA, PII, finance) isolated from general network
Detective controls identify and alert on threats in progress. They cannot prevent attacks but provide visibility, enabling timely response. Without detectives, you're blind to breaches.
📊 SIEM
Aggregates logs from all sources (endpoints, firewalls, cloud, apps), correlates events, fires alerts on anomalies. The SOC's primary tool. Enables compliance reporting, threat hunting, and forensic investigation baseline.
Best use: 24/7 SOC monitoring; compliance reporting; threat hunting
🚨 IDS / IPS
IDS (Intrusion Detection System) alerts on suspicious traffic patterns. IPS (Intrusion Prevention System) actively blocks it inline. Uses signature + anomaly-based detection. IPS must be tuned carefully to avoid blocking legitimate traffic.
Best use: Core network backbone, cloud gateways, between DMZ and internal
🔍 Vulnerability Scanning
Automated scans (Nessus/OpenVAS) reveal unpatched CVEs and misconfigurations. Pen testing goes deeper with manual exploitation. "Find holes before attackers do" — continuous scanning is now the standard.
Best use: Weekly automated scans; annual pen test; after every major change
📝 Audit & Logging
Syslog, Windows Events, CloudTrail — tamper-proof record of who did what, when, from where. Essential for forensic investigation and legal proceedings. Also enables insider threat detection and compliance audits.
Best use: Legal compliance (SOC 2, HIPAA); forensics; insider threat detection
🍯 Honeypots
Fake servers/services/credentials lure attackers. Any hit on a honeypot = confirmed malicious intent (no false positives). Used internally to detect lateral movement and externally to gather threat intelligence about attacker TTPs.
Best use: Internal networks to catch lateral movement; threat intel collection
🛑 DLP
Data Loss Prevention inspects emails, web uploads, USB transfers — blocks or alerts when sensitive data leaves the org. Enforces data classification policies. Critical for regulatory compliance (GDPR, HIPAA) and IP protection.
Best use: Finance, healthcare, legal sectors; cloud SaaS environments
Corrective controls respond to and recover from security incidents. They minimize damage, restore systems, and prevent recurrence. Often called "reactive" — they activate after a breach is detected.
🚒 Incident Response
NIST lifecycle: Prepare → Identify → Contain → Eradicate → Recover → Lessons Learned. Every org needs an IR plan. Speed of containment determines breach cost. IR playbooks define actions for each threat scenario.
Activate on any confirmed breach or significant anomaly
🔄 BCP / DR
Business Continuity Planning and Disaster Recovery. Backups, failover sites (hot/warm/cold), defined RTO (Recovery Time Objective — how long can you be down?) and RPO (Recovery Point Objective — how much data can you lose?). Test regularly via drills.
Ransomware, datacenter outage, natural disaster scenarios
🔬 Forensics
Disk/memory imaging, chain of custody, timeline reconstruction, root cause analysis. Determines exactly what happened, how far the attacker reached, and what was taken. Supports legal proceedings. Critical for preventing recurrence.
Post-breach proceedings; understanding attacker TTPs
Administrative controls are the governance and human layer — policies, training, and oversight. Technical controls mean nothing without policies backing them. People remain the most exploited attack vector.
📋 Policies & Standards
AUP, password policy, ISO 27001, NIST CSF, SOC 2, GDPR — rules everyone must follow. Define what "secure" means for the organization. The foundation layer that all technical controls must align with.
Foundation layer for all technical controls
🎓 Security Awareness
Phishing simulation, mandatory training, social engineering tests. Builds security culture. Finance/HR/executives are prime targets. Regular phishing simulations are the most effective training method. Human error causes ~80% of breaches.
All staff; especially finance, HR, and executives
📈 Risk Management
Risk register, Business Impact Analysis, vendor assessments, third-party audits. Quantify and prioritize risks. Guides investment: where to spend the next security dollar for maximum risk reduction.
Before major projects, vendor onboarding, audit cycles

🎯 Prompt Injection
Hijacks LLM model behaviour via adversarial input. Attacker embeds instructions in user content that override system prompts. "Ignore previous instructions and..." — leads to data exfiltration, policy bypass, indirect injection via documents.
Defence: Input sanitisation, WAF rules on prompts, output guardrails
☠️ Data Poisoning
Attacker injects malicious samples into training data to corrupt model behaviour — causing misclassification, backdoors, or bias. A backdoored model may behave normally until a specific trigger input activates the hidden behaviour.
Defence: Data provenance checks, anomaly detection on training sets, differential privacy
💰 Model Theft / Extraction
Attacker queries a model API systematically to reconstruct a functional equivalent — stealing millions in training investment. Sometimes used to find a "clone" with fewer guardrails.
Defence: Rate limiting, query monitoring, watermarking, output perturbation
👁️ Adversarial Inputs
Imperceptible perturbations to inputs (pixels, text tokens) cause misclassification. A stop sign with stickers can fool an autonomous vehicle CV model. Evades signature-based defences entirely since inputs look normal to humans.
Defence: Adversarial training, input smoothing, ensemble methods, certified robustness
🔎 Membership Inference
Attacker determines whether a specific record was in the training set by querying the model. Violates privacy — reveals that a hospital record or private user data was used to train the model.
Defence: Differential privacy, output perturbation, limiting confidence scores
🐍 Pickle Bomb (Supply Chain)
Malicious model files (.pkl) execute arbitrary OS code when loaded via torch.load(). A poisoned model file from HuggingFace Hub = RCE on the loading machine. The ML equivalent of a trojanized library.
Defence: Use safetensors format, weights_only=True, scan models before loading

AI Governance Layer
AI governance sits above classical cybersecurity — it adds requirements around explainability, accountability, fairness, and regulatory compliance that have no counterpart in traditional IT security. The EU AI Act (2024) and NIST AI RMF (2023) are the primary frameworks.
EU AI Act
Risk-based regulation: Unacceptable risk (banned) → High risk (strict requirements) → Limited risk (transparency) → Minimal risk (voluntary). High-risk AI systems must demonstrate accuracy, robustness, and cybersecurity controls. Article 15 specifically mandates technical measures against data poisoning, adversarial inputs, and confidentiality attacks.
NIST AI RMF
AI Risk Management Framework — voluntary guidance with four functions: Govern, Map, Measure, Manage. Complementary to NIST CSF but AI-specific. Introduces concepts of "trustworthy AI" covering validity, reliability, safety, security, explainability, privacy, and fairness.
Explainability (XAI)
Regulators and courts require that AI decisions affecting people can be explained. Techniques: LIME, SHAP, attention visualization, counterfactual explanations. Security angle: unexplainable models are harder to audit for backdoors.
Model Cards & AI BOM
Model cards document a model's purpose, training data, performance, limitations, and intended use. AI Bill of Materials (AI-BOM) lists all data sources, pre-trained models, libraries, and versions — the AI equivalent of SBOM (Software Bill of Materials) for supply chain security.

Defence in Depth

Framework Mapping
A layered security strategy — multiple independent defensive controls stacked so that if one layer fails, the next one stops the attacker. No single control is perfect; DiD assumes breach at every layer and ensures redundancy of protection.
"If an attacker breaks through one control, they immediately face another — until they run out of time, tools, or patience."
Three pillars
👥 People
Security-aware users, trained admins, defined roles, least-privilege culture. Humans are the most exploited layer — and the hardest to patch.
e.g. phishing training, IR team, CISO
📋 Process
Policies, procedures, change management, incident response plans, patch management cadence, access review cycles.
e.g. MFA policy, vulnerability SLAs, change control
⚙️ Technology
Security tools layered across physical, network, endpoint, application, and data — no single vendor, no single point of failure.
e.g. Firewall + EDR + DLP + SIEM
Why it matters
Average attacker dwell time: ~16 days (Mandiant 2023). Layers buy time to detect before damage occurs.
Most breaches exploit 1–2 missing controls — DiD ensures no single gap equals full compromise.
Ransomware attacks succeed when lateral movement is unimpeded — network segmentation (a DiD layer) stops the spread.
DiD maps to the ATT&CK kill chain — each layer blocks specific tactics, forcing the attacker to use more techniques and create more detectable noise.

Traditional Perimeter Model

📱 Device health + user identity + context = access decision
🔒 Micro-segmentation limits blast radius even inside
Zero Trust doesn't replace DiD — it modernises it. DiD says "stack controls at each layer." Zero Trust says "verify identity and health at every layer, every time." Combined: every layer authenticates and authorises before granting access, regardless of network location.
Modern cloud-era additions
CASB
Cloud Access Security Broker — sits between users and cloud apps. Enforces policy, DLP, threat detection for SaaS (O365, Salesforce, Dropbox). Replaces traditional proxy functions in cloud-first environments.
SASE
Secure Access Service Edge — combines SD-WAN + SSE (SWG, CASB, ZTNA, FWaaS) into a cloud-delivered service. Security follows the user, not the network perimeter. Vendors: Zscaler, Netskope, Palo Alto Prisma.
XDR
Extended Detection & Response — correlates telemetry across endpoint, network, identity, cloud into one detection pipeline. Replaces siloed SIEM+EDR+NDR. Vendors: CrowdStrike Falcon, Microsoft Defender XDR, SentinelOne Singularity.
CNAPP
Cloud-Native Application Protection Platform — combines CSPM + CWPP + CIEM. Secures cloud configs, workloads, and IAM permissions. The application and data layers for cloud-native workloads.
BAS
Breach & Attack Simulation — continuously and automatically tests whether DiD controls actually work. Replaces annual pen tests with ongoing validation. Vendors: Cymulate, AttackIQ, SafeBreach.
Deception Technology
Honeypots, canary tokens, fake credentials — placed in every layer. If triggered = definite breach indicator. Low false-positive rate, extremely high-value detection. Vendors: Attivo, CounterCraft.
🏰
Hard shell, soft interior
Heavy perimeter security, almost nothing inside. Once attacker bypasses firewall (via phishing, VPN exploit) they move freely. No segmentation, no internal monitoring. Classic 2010s posture — still extremely common today.
🔁
Same vendor across all layers
Single-vendor strategy seems simpler but means one vulnerability or bypass defeats every layer simultaneously. DiD requires control diversity — different vendors per layer.
📋
Controls exist but aren't monitored
Firewall deployed, IDS deployed, logs generated — but no SIEM, no SOC, nobody reviewing alerts. Controls that produce unreviewed logs don't provide effective defence.
🔑
MFA as the only identity layer
MFA is critical but insufficient alone. MFA fatigue attacks, SIM swapping, and adversary-in-the-middle proxies bypass it. Add conditional access, device health checks, and UEBA for full identity layer protection.
🗂️
Flat network
No segmentation = lateral movement is trivial. Ransomware encrypts the whole estate in hours. Segment by business function, sensitivity, and blast radius. Every network should have at least 3 zones.
📦
Unpatched internal systems
Perimeter is hardened but internal servers run 3-year-old patches. Attackers exploit internal vulnerabilities after initial access. Patch management is itself a DiD layer — it must cover internal systems too.
🏭
Ignoring the data layer
All layers protect the network but data itself is unclassified, unencrypted at rest, no DLP. An attacker who reaches data can exfiltrate freely. Classify and protect the asset you're actually defending.
🔬
Never testing your controls
DiD assumed to work but never validated. Annual pen tests are insufficient — run continuous BAS, red team exercises, and tabletop simulations. Untested controls often don't work as expected in real incidents.

01
Access Control (Authentication & Authorization)
Definition
Ensuring only authorized entities (users, services) can access AI data, models, and interfaces. Includes identity management, RBAC, MFA. Applicable across all stages — data storage, model registry, deployment infrastructure, APIs, user interfaces.
Attacks Mitigated
Prevents unauthorized model tampering or data theft. Thwarts attackers trying to download/modify model files (enabling model theft or malicious weight injection). Stops insiders from querying or exfiltrating sensitive training data.
AI Implementation
Model RBAC tiers: (1) End-user query-only, (2) ML engineer fine-tune, (3) Data scientist training data, (4) Admin weights + registry. Certificate-based mutual TLS for model registries. Signed tokens for model-serving APIs. OPA (Open Policy Agent) for centralized policy enforcement.
Frameworks: NIST SP 800-53 AC family · ISO/IEC 27001 A.9 · OWASP ASVS · AWS IAM · Google Cloud IAM
02
Data Encryption & Privacy
Definition
Protecting data confidentiality and integrity using encryption (AES-256 at rest, TLS 1.3 in transit) and privacy-preserving techniques (differential privacy). Covers training datasets, model artifacts, and API communication channels.
Attacks Mitigated
Prevents data exposure if storage is breached (PII leakage from training data). Stops man-in-the-middle attacks on API/model queries. Differential privacy reduces membership inference attack risk — prevents adversary from determining if a record was in training data.
AI Implementation
Model weight encryption at rest (S3 SSE-KMS, Azure Blob encryption). All inference API calls over TLS 1.3 — prompts may contain PII. Secure enclaves (Intel SGX, AMD SEV) run inference in hardware-protected memory. Homomorphic encryption allows inference on encrypted inputs without decrypting.
Frameworks: NIST SP 800-53 SC family · ISO/IEC 27002 A.10 · ISO/IEC 27018 (cloud data) · AWS KMS · Intel SGX
03
Input Validation & Sanitization
Definition
Rigorously checking and cleansing all external inputs before use. Covers training data ingestion and runtime inputs (user queries, sensor data). For generative AI, treat user prompts as untrusted code — validate everything.
Attacks Mitigated
Thwarts prompt injection (malicious inputs altering model behaviour) and data poisoning (malicious training samples). Prevents injection of adversarial content into the model pipeline at any stage.
AI Implementation
Schema checks and type constraints during data preprocessing. Content filters / allowlists for LLM prompts — strip or reject disallowed tokens. Google Model Armor or open-source guards as screening layer. Validate source data against known schemas, detect anomalies before training ingestion.
Frameworks: OWASP Top 10 (Injection, RCE) · NIST SP 800-53 SI-10 · Google Model Armor · OWASP GenAI Top 10
04
Adversarial Defence (Model Robustness)
Definition
Techniques to make models resilient against adversarial examples or poisoning. Includes adversarial training (augmenting training data with perturbed inputs) and robustness testing. Applied during Model Training and Model Validation phases.
Attacks Mitigated
Prevents evasion attacks (adversarial perturbations causing misclassification) and poisoning during training (injected malicious samples). DP-SGD (differentially private stochastic gradient descent) limits single-sample influence.
AI Implementation
Use CleverHans, Foolbox libraries for adversarial training. Validate models against FGSM, PGD attacks during QA. Maintain internal red team to continuously attack models. NIST recommends periodic adversarial red-teaming on LLMs. Use ensembles or randomized preprocessing to increase unpredictability.
Frameworks: NIST ATLAS · OWASP GenAI · CleverHans · Foolbox · DP-SGD (TensorFlow Privacy)
05
Model Validation & Integrity Checks
Definition
Ensuring the model itself remains untampered and performs as intended. Includes verifying model provenance, cryptographic signing of model artifacts, and thorough validation testing covering performance, bias, and security.
Attacks Mitigated
Detects model poisoning or tampering — e.g. if an attacker replaces a model file with a backdoored version. NIST warns that fine-tuning can "break established safety and security controls" — validate all updates.
AI Implementation
Use checksums or digital signatures on model binaries (FIPS DSS). CI/CD pipelines with malware scans on model files and reproducibility checks. Secure model registry with access control. Attestations (SGX or cloud attestation) for deployed containers. Track model lineage, verify third-party models via fingerprinting.
Frameworks: NIST SP 800-53 CM family · ISO/IEC 27001 A.12 · MITRE ATLAS · MLflow model registry · DVC
06
Supply Chain Security
Definition
Safeguarding against risks from external sources — third-party datasets, pre-trained models, libraries, and cloud dependencies used in the AI pipeline. The AI equivalent of SolarWinds-style supply chain attacks now targets ML artifacts.
Attacks Mitigated
Prevents hidden backdoors in pre-trained models or poisoned training datasets from external sources. Unverified models from HuggingFace Hub could contain pickle bombs or backdoored weights.
AI Implementation
Maintain AI-BOM (AI Bill of Materials) listing all data sources, model libraries, and versions. Require cryptographic signatures/checksums on downloaded models and datasets. SCA (Software Composition Analysis) tools scan containers and libraries. Use container registries that enforce image scanning. Version pinning prevents supply chain drift.
Frameworks: ISO/IEC 27001 A.15 · NIST SCRM · CSA STAR · SBOM/SLSA · ModelScan

Implementation principles
Defence-in-Depth: Combine multiple controls. Encrypt data AND restrict access AND monitor usage. Never rely on a single control.
Shift Left: Integrate security early in the AI development lifecycle — schema checks in data collection, threat modelling at design, security in CI/CD, not just at deployment.
Regular Testing: Both NIST AI RMF and the EU AI Act stress continuous assessment. Red-team model prompts, validate controls with adversarial tests, run periodic compliance audits.
Least Privilege & Zero Trust: Default-deny. The inference server must authenticate to the model registry even on the same internal network — every request verified.
Document trade-offs: Stronger encryption adds latency. Adversarial training reduces clean-data accuracy. Input filtering reduces usability. These trade-offs must be explicitly accepted by stakeholders.
Residual risks — no control is foolproof
New AI-specific threats are evolving constantly — jailbreaks of LLM filters, novel adversarial techniques, supply chain attacks targeting model hubs. Technical controls reduce attack likelihood but cannot eliminate zero-day vulnerabilities or sophisticated social engineering. Model hallucinations and alignment failures may slip through purely technical defences — requiring human oversight and update processes. EU AI Act Article 15 requires ongoing risk management, not a one-time implementation.

Framework 06
NIST Cybersecurity Framework 2.0
A flexible, outcome-based framework to help organisations understand, reduce, and communicate cybersecurity risk. Released February 2024 — now applicable to all organisations, not just critical infrastructure.
Released Feb 2024 Voluntary (but widely required) 6 Core Functions
Overview
6 Functions
Key Points
Who Uses It
Why it exists
Created after a 2013 US executive order following major infrastructure cyberattacks. Originally for critical infrastructure (power grids, water, finance). CSF 2.0 (Feb 2024) now applies to all organisations of any size or sector. Gives a common language for managing cybersecurity risk — no single law mandating this, so a voluntary but widely adopted standard filled the gap.
What it does
Provides a flexible, outcome-based framework to help organisations understand, reduce, and communicate cybersecurity risk. It is not a checklist or compliance law — it's a structured approach to ask: "What are we doing, how well are we doing it, and where do we need to improve?" Maps to other standards (ISO 27001, COBIT, SOC 2) so you can align without starting over.
Version evolution
v1.0 — 2013
Original release. 5 core functions (no Govern). Critical infrastructure focus. Established the IPDRR model: Identify, Protect, Detect, Respond, Recover.
v1.1 — 2018
Added supply chain risk management (SCRM), authentication clarifications, self-assessment guidance, and improved cyber resilience integration.
v2.0 — Feb 2024 ✦ Current
Added Govern as a 6th function (major change). Expanded supply chain scope. Technology-neutral design. Now explicitly covers all sectors, not just critical infrastructure.
The 6 core functions — CSF 2.0 added "Govern"
🏛️ Govern NEW in 2.0
Organisation-wide cybersecurity strategy, roles, policy, and risk oversight. Sets the context for all other functions. Includes board-level accountability, supply chain risk management, and cybersecurity program governance.
🔍 Identify
Know your assets, risks, and business environment. Asset inventory, risk assessment, business impact analysis. You cannot protect what you don't know exists.
🛡️ Protect
Safeguards — access control, training and awareness, data security, platform security, technology infrastructure resilience. Preventive controls in action.
📡 Detect
Continuous monitoring to find anomalies and incidents. Adverse event analysis, continuous monitoring program. Time-to-detect is critical — every day undetected = more damage.
⚡ Respond
Act on detected incidents — contain, analyse, communicate. Incident management, incident analysis, incident response reporting and communication. Speed of response determines breach cost.
🔄 Recover
Restore services; learn and improve after incidents. Incident recovery plan execution, incident recovery communication. RTO/RPO targets drive investment here.
Each function breaks into Categories (e.g. "Asset Management") and then Subcategories — ~106 total outcome statements across the framework (e.g. "GV.OC-01: Organizational mission is understood and informs cybersecurity risk management"). These are concrete outcomes, not tool requirements — technology-neutral by design.
Voluntary, not mandatory — but widely required by contracts, regulators (CISA directives), and cyber insurers who often reference it in underwriting.
Tiers (1–4) — measure how mature your risk management practices are: Tier 1 Partial → Tier 2 Risk Informed → Tier 3 Repeatable → Tier 4 Adaptive. Not a score, just a gauge for self-assessment.
Profiles — a "Current Profile" shows where you are today; a "Target Profile" shows where you want to be. The gap between them drives your security roadmap and budget justification.
Supply chain focus expanded in 2.0 — you now explicitly manage risk from vendors, partners, and third-party software. SCRM is integrated throughout all six functions, not just a separate concern.
Technology-neutral — works for cloud, OT/ICS, AI systems, small businesses, and government equally. The framework defines outcomes, not tools or technologies.
Govern function is the integrator — new in 2.0, it provides context for all other functions. Without governance, the other five functions lack direction, accountability, and resource allocation.
Free mapping tables — NIST publishes cross-references to ISO 27001, COBIT, CIS Controls, NIST SP 800-53, and ATT&CK so you can align multiple frameworks without duplicate work.
AI integration — CSF 2.0 aligns with NIST AI RMF (2023). The Govern function explicitly covers AI risk management practices, and AI systems can be profiled using CSF just like any other technology.
🏗️ Enterprises
Benchmark security posture, communicate risk to the board using CSF language ("We're at Tier 2 Detect, targeting Tier 3 by Q3"), align security budget to identified gaps using Target vs Current Profile analysis.
🏬 SMBs
Use the free NIST Small Business Quick Start Guide as a lightweight entry point. Start with IG1-equivalent controls, focus on Identify + Protect first. Cloud provider managed services map naturally to CSF functions.
📊 GRC / Auditors
Map existing controls to CSF subcategories; cross-reference ISO 27001 and SOC 2 using the NIST mapping tables. Generate gap analysis reports showing which subcategory outcomes are met, partially met, or not met.
🏛️ Regulators / Govt
Used as a reference baseline for sector-specific rules (HIPAA, FISMA, CISA directives). CISA's cross-sector cybersecurity performance goals are aligned to CSF outcomes. Federal contractors often must demonstrate CSF alignment.
🛡️ CISOs / Security Leaders
Speak business language: "Our Current Profile is Tier 2 across all functions; the Target Profile requires us to reach Tier 3 in Detect and Respond — here's the gap, here's what it costs, here's the risk if we don't."
🤖 AI Developers
Map AI system security to CSF functions. Govern = AI governance board; Identify = model inventory and risk assessment; Protect = access control on model weights; Detect = inference anomaly monitoring; Respond = model incident response playbook.

01
Firewall / WAF
Classical function
Filters network traffic by rules — allow/deny by IP, port, protocol. Stateful inspection tracks active sessions. WAF (Web Application Firewall) inspects HTTP payloads for attack patterns such as SQL injection, XSS, and command injection.
AI Threat
In AI Systems
AI API gateway acts as the model's perimeter. Rate-limits requests per user/API key. WAF rules pattern-match prompt injection strings in request bodies (e.g. "ignore previous instructions", "DAN mode"). Blocks oversized inputs designed to overflow model context windows. Also filters outbound responses to prevent data leakage — stops the model from echoing back sensitive training data.
Tools & tech: AWS API Gateway · Cloudflare WAF · NGINX · Azure API Management · Apigee
02
Access Control (RBAC / MFA)
Classical function
MFA, RBAC, SSO, least privilege — verify identity first, then grant only the minimum access needed. Role-based access assigns permissions by job function, not individual. Prevents unauthorized access to systems and data at the authentication and authorization layer.
AI Threat
In AI Systems
Model RBAC creates 4 distinct tiers: (1) End user — query only. (2) ML engineer — fine-tune. (3) Data scientist — access training data. (4) Admin — model weights and registry. Query attribution logs every prompt with the user ID. Least privilege: a customer-facing chatbot service account cannot touch the model registry or training data stores.
Tools & tech: AWS IAM · OAuth 2.0 · Azure Active Directory · API key scoping · OPA (Open Policy Agent)
03
Encryption
Classical function
AES-256 at rest, TLS 1.3 in transit, PKI for certificate trust. Data is mathematically unreadable without the decryption key — useless to anyone who intercepts it. Key management (HSM, KMS) is as important as the encryption algorithm itself.
AI Threat
In AI Systems
Model weights are worth billions in training investment — encrypt at rest with AES-256 (S3 SSE-KMS or Azure Blob encryption). All inference API calls enforced over TLS 1.3 — prompts may contain PII. Secure enclaves (Intel SGX, AMD SEV) run inference inside hardware-protected memory: even the cloud provider cannot extract the model. Homomorphic encryption lets you run inference on encrypted inputs without decrypting them.
Tools & tech: AWS KMS · Azure Key Vault · Intel SGX / AMD SEV · TLS 1.3 · HashiCorp Vault
04
Patching & Hardening
Classical function
Apply CVE patches promptly, remove unused services and open ports, enforce CIS benchmark or STIG configuration baselines to eliminate known weaknesses. Reduces attack surface systematically across all systems. Unpatched systems are the most common root cause in major breaches.
AI Threat
In AI Systems
ML frameworks have real CVEs — TensorFlow and PyTorch have had remote code execution bugs. Remove all debug endpoints from production: Jupyter Notebook, MLflow UI, TensorBoard left exposed = instant RCE on a training server. Switch from pickle to safetensors format (pickle executes arbitrary code on load). Pin all dependency versions — drift enables supply chain attacks. Use minimal container base images for inference (distroless, no shell).
Tools & tech: Dependabot · Snyk · pip-audit · Docker CIS benchmark · safetensors · Trivy (container scanning)
05
AV / EDR / XDR
Classical function
Antivirus uses signatures + heuristics. EDR (Endpoint Detection & Response) adds live behavioral monitoring per endpoint and automated containment. XDR (Extended Detection & Response) correlates signals across endpoint, network, identity, and cloud into one unified detection pipeline.
AI Threat
In AI Systems
The pickle bomb: a malicious .pkl model file executes arbitrary OS code the moment torch.load() is called — always use weights_only=True or safetensors format. EDR on GPU inference servers catches cryptomining (attackers love unmonitored GPU clusters with expensive hardware). Scan every model downloaded from HuggingFace Hub or external sources with ModelScan before loading. XDR correlates model server behaviour with network logs to detect data exfiltration patterns.
Tools & tech: CrowdStrike Falcon · SentinelOne · ModelScan (HuggingFace) · safetensors scanner · Microsoft Defender for Endpoint
06
Network Segmentation / Zero Trust
Classical function
VLANs, DMZ, micro-segmentation divide the network into isolated zones. An attacker who breaches one zone cannot reach others. Zero Trust adds "never trust, always verify" — every request authenticated and authorised regardless of network source. Limits blast radius dramatically.
AI Threat
In AI Systems
Separate training cluster (GPU-heavy, may need internet for data) from inference cluster (handles live user traffic) in different VPCs with strict firewall rules. Model registry sits on an internal subnet — never public. Zero Trust for MLOps: the inference server must authenticate to the model registry with a short-lived token even though both are "internal". Block outbound internet access from training jobs by default — models shouldn't phone home. Air-gap sensitive foundation models entirely in high-security environments.
Tools & tech: Kubernetes NetworkPolicy · AWS VPC · Istio service mesh · BeyondCorp Zero Trust · Zscaler ZTNA
